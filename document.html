00

概览与设计哲学

Core

FemWA（Flow Engine for Minds · Work Automata） 是一套用于编排多 Agent 工作流的 DSL（领域特定语言）+ 编译器。它的核心理念是：用写剧本的方式描述 AI 协作流程。

💡
三条核心原则，理解这三点就理解了 Fem 的设计：
① Scope = 空间：在同一 scope 的角色才共享上下文，天然视角隔离；
② Vars = 流程控制：变量不是用来传消息的，是用来控制流程走向的；
③ 上下文不是变量：LLM 之间的沟通靠共享上下文，非必要不传参。

Fem 文件扩展名为 .fems（script 的 s）。编译器读入 .fems 文件，解析剧本，按照规则驱动 Agent 协作。

📖
本文档对应 FemWA v1.0 编译器行为。文中 TODO / 暂未实现 的特性已标注，请注意区分。
01

文件结构

Core

一个 .fems 剧本按以下顺序组织，各块均可选（除 mainflow 必须存在）：

fems · 整体结构
meta:       # 剧本元信息
vars:       # 可选 · 全局状态变量池 · 如有变量，须先声明再使用
code:       # 可选 · 外部 Python 模块引入 · 如果需要@func 功能
actors:     # 应有 · 角色定义 · 如无就调用无角色信息的AI
memory:     # 可选 · 记忆检索方法定义
context:    # 可选 · 上下文提取方法定义
action ...: # 必须 · 动作定义 · 至少一个，可以多个
module ...: # 可选 · 模块定义 · 可以多个
mainflow:   # 必须 · 主流程编排
⚠️
缩进敏感：缩进表示从属关系，类似 Python。换行敏感：在 fork、for、par 等流程控制语法中必须严格按规定格式换行缩进，否则解析器报错。
02

基本约定 & 符号速查

Core
  • 大小写敏感：变量名、action 名区分大小写
  • 注释：行内使用 # 或 //
  • @ 符号：Actor 实体名字永远带 @，定义、引用、传参均须保留
  • 文件路径：必须用 file:"path/to/file" 格式，裸文本为字面量字符串
  • 多行文本：用 | 后换行缩进表示多行字面量

核心符号

符号语义示例
action定义动作（类似 Python def）action wolf_kill @ai(@wolf):
@引用/指向 Actor 实体@wolfClaire、@ellis.type
[]Flow 节点位置标记[A]、[START]、[END]
{}变量插值（prompt f-string）alive_players
<<>>AI 输出赋值信号SET VARIABLE: <<@VOTE = @player3>>
&引用 Module（黑箱）&CoderSandbox(arg)
->Flow 连接方向[A] -> [B]

中文符号等价表

为方便中文输入，以下中文符号在 Fem 中与英文等价：

中文符号等价英文备注
：:冒号
，,逗号
" ""引号
（）()圆括号
【】[]方括号
｜|竖线
--->仅在 flow 区域等价，prompt 中不受影响
《》〈〉<<>>仅在 AI 输出赋值识别中等价
03

meta 元信息

Core

定义剧本的基础属性与运行环境。

fems
meta:
    name    = "我的狼人杀"
    version = 1.0
    owner   = [1, 2]
    database = file:"werewolf.db"
    session = new           # 或指定整数 session_id
    system_safety = |
        不许删库，不许生成危险内容。
    output_style = "回复请简洁专业"
字段类型说明
name文本剧本名称，新建 session 时写入数据库
owner整数数组所有者的 user_id，编译器自动注入用户信息到上下文
database文件路径SQLite 数据库路径，未指定默认 ./chronica.wor
session整数 / newnew 自动新建；写数字则加载指定 session；也可写 17/new
system_safety文本 / 文件路径安全须知，自动注入所有角色的上下文
output_style文本 / 文件路径输出质量要求，自动注入上下文
ℹ️
session = 17 / new 的含义：如果 session 17 已存在就加载，如果新建的恰好是 17 则自动新建，否则后端会询问是否新建。
04

vars 全局变量池

Core

所有全局状态变量都在此声明。变量必须先声明才能使用，未声明直接赋值会报错。

fems
vars:
    turn_count     = 1
    game_over      = false
    alive_players  = [@seer, @wolf]        # @actor 数组
    agent_locations = {@ellis: "卧室", @bob: "公园"}  # 字典
    @speaker        = ""                 # @actor 类型变量，空值用 ""
    hp             = {@wolf: 100, @seer: 100}

要点

  • @actor 类型变量：变量名必须以 @ 开头，如 @speaker，才能被识别为 actor 类型
  • @actor 双视角访问：hp.@wolf 与 @wolf.hp 完全等价（见第 19 节）
  • 字典缺省值：prompt 中 {dict[key]} 若 key 不存在，解析为空字符串，不报错
05

actors 角色定义

Core

采用 类型 @名字 = 属性 的强类型声明风格。

fems
actors:
    ai @hostgod = soul:0, source:deepseek
    ai @wolf    = soul:1, source:glm5.1, tools:[deep_think]
    ai @seer    = soul:2, source:glm5
    human @player = soul:9, source:0  # source:0 = 无管理员视角的人类用户
字段说明
soul:ID唯一灵魂 ID，对应数据库 souls 表，编译器自动提取系统提示词
source:模型名AI 写模型名；人类写数字对应 user 表；可缺省随机分配
source:0特殊值：无管理员视角的人类用户，只拥有 soul 的视角（适合作为狼人杀玩家）
tools:[...]挂载工具列表，如 web_search、deep_think
🚫
Actor 实体名字永远带 @，任何地方都不得去掉：定义时 ai @ellis = ...、引用时 @ai(@ellis)、变量传递时 @speaker = @ellis。@ 是变量名的一部分，去掉就无法识别为 actor 类型。
06

code 外部 Python 模块

Core

引入 Python 文件，供 @func 和 resolve 调用。路径必须使用 file:"..." 格式。路径须为绝对路径；若写相对路径如 file:"wait.py"，则指向 /Users/{user_name}/FemWA/func_code/wait.py。

fems
code:
    game_logic = file:"utils/game.py"
    dev_ops    = file:"utils/deploy.py"
    memfile    = file:"femBridges/MemoryExample.py"
⚠️
以下名称为保留字，不可用作 code 别名：meta、vars、code、actors、action、module、flow、mainflow、memory、context
07

memory & context 定义

Core

memory 和 context 方法用于自定义如何向 AI 提供记忆和上下文。它们与 action/module 平级。

fems · 定义语法
memory rag10(memfile.retrieve_example):
    in: prompt, session_id, @actor
    out: memory_text

context thisSession(ctxfile.findThisSession):
    in: session, @actor
    out: context_text

in 参数预留字段

字段名传入值
prompt当前 action 的 prompt（已完成变量替换）
@actor当前说话者的 actor_info 字典（Python 侧参数名自动映射为 actor_info）
session / session_id当前 session ID
turn / turn_id当前 turn ID
其他名称从全局 vars 中取同名变量

在 action 中引用

fems
action speak @ai(@ellis):
    prompt: "你好"
    memory:  rag10        # 不写则用第一个 memory 方法；无定义则用内置默认
    context: thisSession  # 不写则用第一个 context 方法；无定义则用内置默认
ℹ️
如果未定义任何 context 方法，FEM 编译器自带默认实现——按 scope 检索当前 session 的上下文。大多数场景直接用内置默认即可。
08

AI 动作 @ai

Core

最常用的动作类型，调用 LLM 完成一轮对话。

fems · 完整字段
action wolf_kill @ai(@wolf):
    prompt: |
        现在是第{turn_count}天夜晚，存活玩家：{alive_players}。
        你是狼人，请选择击杀目标。
        分析后输出：SET VARIABLE: <<@KILL = @玩家名>>
    scope: [@hostgod, @wolf]
    out: wolf_target(string, "击杀目标")
    resolve: game_logic.resolve_target(@KILL, alive_players)
    max_retries: 3
    fallback: host_emergency
    context: thisSession
    memory: rag10

字段详解

prompt

支持 {var} f-string 插值。默认自动替换所有全局变量；如果写了 in: 字段，则只替换 in 中列出的变量（用于控制信息暴露范围）。

fems · in 字段示例
action coder_act @ai(@coder):
    in: my_task = task_list[@coder]   # 左边=prompt里用的名字，右边=全局变量表达式
    prompt: "你的任务是：{my_task}"    # {other_var} 不会被替换，原文保留

scope

定义"哪些角色能看到这轮对话记录"。支持动态变量，如 scope: playersInPark。详见第 18 节。

out — AI 输出变量

声明需要 AI 输出的变量。格式：变量名(类型, "标签")。AI 必须在回复中用 SET VARIABLE: <<变量名 = 值>> 赋值。变量须已在 vars 中声明。

支持写入字典键：vote_results.@voter(string, "")，将值存入字典的指定键。

resolve — 校验函数（可选）

当需要对 AI 输出进行复杂校验或兼容非标准输出格式时使用。

fems + python · resolve 示例
# .fems 中
resolve: game_logic.resolve_target(@KILL, alive_players, SET_VARIABLE)
# 函数中写了 SET_VARIABLE 时，编译器会传入解析失败的赋值列表

# Python 中（utils/game.py）
def resolve_target(@KILL=None, alive_players=None, SET_VARIABLE=None, **kwargs):
    if @KILL and @KILL in alive_players:
        return [(True, False, "")]          # (is_success, show_to_ai, feedback)
    if SET_VARIABLE:                          # 内置解析失败，自己处理
        import re
        for item in SET_VARIABLE:
            m = re.match(r'@KILL\s*=\s*@(\w+)', item)
            if m and f"@{m.group(1)}" in alive_players:
                return [(True, False, "")]
    return [(False, True, "目标不在存活玩家中，请重新选择。")]
⚠️
当前版本限制：max_retries 和 fallback 字段已预留但未实现自动重试，resolve 返回值中 show_to_ai 和 feedback 也暂未生效。

动态 @actor_expr

ℹ️
当 action 括号里使用动态变量时（如 @ai(@speaker)），for/par 的循环变量名必须与括号里的变量名完全一致，否则引擎直接报错。
09

人类动作 @human

Core

语法与 @ai 基本一致，用于将人类玩家的操作纳入流程。

fems
action human_vote @human(@player):
    prompt: 请投票：
    scope: [@hostgod] + alive_players
    out: vote_target(dropdown, choices=alive_players, label="投票")
    resolve: game_logic.resolve_human_vote

# 让人类扮演特定 AI 角色，继承该角色的 scope：
action seer_act @human(@player) as (@seer):
    prompt: 你现在是预言家，你要查验谁？
    scope: [@seer, @hostgod]
  • out 的 dropdown 类型会映射为前端 UI 下拉组件
  • 人类动作有上下文（context 写法与 AI 相同），无需 memory
  • as (@角色)：让人类玩家进入指定 AI 角色的私密视角
10

赋值动作 @assign

Core

轻量级赋值，不调用 LLM，直接修改全局变量。

fems · 支持的操作
action next_turn @assign:
    out: day_count += 1

action reset_scene @assign:
    out: current_scene = "酒馆", time = sunset   # 多个赋值逗号分隔

action update_park @assign:
    out: playerInPark   = add(@Alice)
         playerInMarket = remove(@Alice)   # 多行 out 支持
操作含义
= 值直接赋值
+= N加 N
-= N减 N
= add(元素)列表追加元素
= remove(元素)列表移除元素
11

Python 函数调用 @func

Core

调用 code: 区域声明的 Python 函数，处理复杂逻辑。

fems
action resolve_night @func(my_utils.resolve_night):
    in:
        kill_target = wolf_target        # 左=Python参数名，右=FEM变量表达式
        save        = witch_save_target
    out: dead_tonight, alive_players

in 传参规则

  • 显式模式（写了 in:）：严格按声明传参，格式 参数名 = 变量表达式
  • 自动推断模式（不写 in:）：根据 Python 函数签名，去 vars 中找同名变量传入

out 接收规则

函数返回类型out 行为
返回 字典按 key 匹配 out 中的变量名，key 必须与 out 变量名完全一致
返回单值out 恰好声明一个变量时，直接赋值
out: dict.key 形式函数只返回最终值，编译器负责写入字典的指定键

@actor 变量传给 Python

当 in 中的变量是 @actor 类型时，引擎自动打包为字典传给 Python：

python · 接收到的 actor 字典
{
    "type":     "ai",
    "name":     "@ellis",
    "soul":     3,
    "hp":       100,         # 来自 vars 中 hp 字典
    "location": "酒馆"       # 来自 vars 中 location 字典
}
⚠️
常见错误：out: dict.key 时函数应只返回最终值，不要自己组装字典；返回字典时 key 必须与 out 变量名一一对应，多余或缺失都会报错。
12

基础链式流程

Flow

Flow 将 Node（节点位置）串起来。[START] 和 [END] 是保留关键字，mainflow 中必须存在。

💡
核心区分：Action = 行为（做什么，无位置）；Node = 位置容器（装着 action 或 module）；Flow = 把 Node 串起来的有向图。

引用 Action 无前缀；引用 Module 加 & 前缀。
fems · 顺序链
[START] -> action0 -> [B]:action1 -> &module1 -> [END]

# 太长时可拆成多行，首尾节点接上即可：
[START] -> wolf_discuss -> seer_check -> [A]:tell_seer
[A] -> witch_ask -> resolve_night -> [END]

循环必须用节点

❌ 不是循环
# 只是又执行了一遍 action1
action1 -> action2 -> action1
✅ 真正的循环
# 回到 [A] 的位置，继续跑后续
[A]:action1 -> action2 -> [A]
13

并发分支 fork

Flow

fork 声明此处有并发分支，每条分支运行在新的子线程中，适合同时调用多个 LLM。

fems · 两种写法
# 写法 1：缩进分支
[A] -> [B] ->
fork:
    -> [visit]
    -> [solo]

# 写法 2：同缩进多行（等价）
[C] -> [visit]
[C] -> [solo]

# 带条件 if（Python 表达式，@actor 自动转字典求值）
[A] ->
fork:
    -> if (var == true) -> [B]
    -> if (score > 5 and level >= 3) -> [C]
    -> if (@Portia.hp == 0) -> [D]   # @actor 表达式直接用
ℹ️
fork 只管分支，不管合并。合并用 join（见下一节）。无 if 的分支无条件全部并行启动。
💡
实际上可以完全不写 fork，直接用 mermaid 风格的多行箭头描述分支，有时更简单直观。用 fork 的好处：分支多时少打字，结构整齐便于 debug。

14

合并 join

Flow

join 在接收端定义，声明"我要等哪些上游节点"。

fems
join(all):
    [B] ->
    [C] ->
to [next_node]

# join(2)：等到 2 个上游到达就继续，其余分支直接关闭
join(2):
    [D] ->
    [E] ->
    [F] ->
to [next_node]
  • join(all)：等所有声明的上游节点都到齐
  • join(N)：等到 N 个上游到达，关闭其余分支（只关闭本次 fork 的任务）

join 与 mermaid 写法等价

fems · 两种写法等价
# join 写法
join(all):
    [B] ->
    [C] ->
to [next_node]

# 等价的 mermaid 直接写法
[B] -> [next_node]
[C] -> [next_node]
💡
也可以不写 join，用 mermaid 风格的多行写法指向同一个节点，便自然汇聚——等价于 join(all)。只有写 join(N) 时才能实现"来够 N 个就停掉其余分支"的需求。
15

串行循环 for

Flow

遍历数组，对每个元素依次（串行）执行循环体，等价于展开的顺序链。整个 for 块是一个独立单元，前置节点写在同一行的 for 关键字之前。

fems · 完整 for 块
# 完整结构：前置节点写在 for 同一行
[B] -> for @speaker in speaker_array:
    -> [C]:speak -> [D]:respond ->
-> [END]

# 循环变量不限于 @actor，普通变量也可以
[A] -> for i in [1, 2, 3]:
    -> tick_action ->
-> [END]

# for + if：按类型路由（出口节点可省略）
[B] -> for @player in alive_players:
    -> if (@player.type == "ai")    -> ai_speak ->
    -> if (@player.type == "human") -> human_speak ->
-> [D]
💡
始末节点标记：循环体内，始节点前加 -> 在行首，末节点后加 -> 在行尾。出口节点可有可无——有则循环结束后走向它，无则本分支自动停止。去掉这些行首行尾箭头后，循环体内部语法与外部 flow 完全一致。
🚫
循环变量若是 @actor 类型，变量名必须与 action 定义括号里的动态变量名完全一致，否则引擎直接报错（以防静默用错人）。

动画演示：for 串行循环

16

并发 par

Flow

fork + join 的懒人写法：遍历数组，对每个元素并发执行。极大提升并行调用 LLM 的速度。前置节点同样写在 par 关键字同一行。

fems · par 示例
# 示例1：前置节点在同一行，并发调用模块
[A] -> par @coder in coders:
    -> &CoderSandbox(@coder, task_list[@coder]) ->
-> [D]

# 示例2：每人并发执行多步
[before] -> par @coder in workers:
    -> [A]:writecode -> [B]:trycode ->
-> [D]

# 变量不限于 @actor
[A] -> par i in [1, 2, 3]:
    -> tick_action ->
-> [END]

# 出口节点可省略，各分支结束后自动停止
[VOTE] -> par @player in alive_players:
    -> cast_vote ->
# 无出口节点，各分支结束后自动停止
  • 分支内部多个 node 串行执行，分支之间并发
  • 并发参数是分支局部变量，不同分支取值不同，互不干扰；action 内直接引用，无需显式传参
  • 始末节点标记方式与 for 相同：行首 -> 标始节点，行尾 -> 标末节点
  • 出口节点可有可无——有则各分支汇合后走向它，无则各分支自动停止
  • 并发变量不限于 @actor，par i in [1,2,3]: 也完全合法
  • 并发变量若是 @actor 类型，变量名必须与 action 定义括号里的动态变量名完全一致，否则引擎报错

动画演示：par 并发执行

17

模块 module

Advanced

Module 是包含子流程的黑盒，有入口（[IN]）、出口（[OUT]）、中断出口（[BREAK]）、内部 vars 和内部 flow。

fems · 模块定义
module CoderSandbox(task_var):
    vars:
        finish = false         # 局部变量，离开模块后清空
    
    action write_code @ai(@coder):
        prompt: "任务：{task_var}，请调用 shell 工具编写代码。"
        scope: [@coder]
    
    action submit_code @ai(@coder):
        prompt: |
            测试代码，确认没问题后输出：
            SET VARIABLE: <<finish = true>>
        out: finish(bool, "结束了没")
    
    flow:
        [IN] -> write_code -> submit_code ->
        fork:
            -> if (finish == true)  [OUT]
            -> if (finish == false) [IN]

scope 视角隔离

Advanced

scope 定义"哪些角色能看到这轮对话记录"，是 Fem 视角隔离的核心机制。

fems · scope 用法
# 静态 scope：只有这两个角色能看到
scope: [@hostgod, @wolf]

# 动态 scope：变量在 action 执行时实时解析为 @actor 列表
scope: playersInPark

# 混合
scope: [@hostgod] + alive_players

系统存储聊天记录时，会同时存储本条 scope 中的 soul id 到数据库的 soul_scope 字段。FEM 内置的 context 方法默认通过此字段检索，实现各角色只能看到自己亲历的上下文。

💡
实用比喻：scope 像一个"房间"——只有在场的人才能记得这段对话。更准确的说：神把所有历史都存进宇宙硬盘，但只有亲身经历的人有资格检索。这就是狼人杀视角隔离只需一行 scope 的原因。
19

@actor 类型系统

Advanced

Actor 不仅是角色配置，更是一种变量类型，拥有身份和状态，可以像变量一样赋值、传递。

双视角属性访问

字典视角与实体视角完全等价，可随语境自由选用：

fems · 等价写法
# 假设 vars 中有：hp = {@wolf: 100, @seer: 100}

hp.@wolf        # 字典视角：在 hp 花名册里翻到 @wolf
@wolf.hp        # 实体视角：@wolf 这个人的 hp
# 完全等价，prompt 中、赋值中、if 条件中均可随意使用

# prompt f-string
"你的当前血量是 {hp.@wolf}"      # 同
"你的当前血量是 {@wolf.hp}"

# AI 赋值信号
SET VARIABLE: <<hp.@wolf -= 30>>   # 同
SET VARIABLE: <<@wolf.hp -= 30>>

静态保留属性

解析 @实体.属性 时，先检查以下 5 个保留属性，未命中则去 vars 查同名字典：

type
角色类型："ai" 或 "human"
soul
灵魂 ID（整数）
source
模型名或用户 ID
tools
挂载工具列表
name
角色名字符串

这意味着你无需修改 actors 定义即可给角色挂载动态状态——只需在 vars 中建立对应字典，@实体.属性 自动生效。

AI 变量赋值信号

Advanced

通过在 prompt 中引导，让 AI 在回复中输出赋值信号，驱动流程控制变量的更新。

AI 输出示例（多种合法格式）
# 英文格式
我投票给3号。SET VARIABLE: <<@VOTE_TARGET = @player3>>
我觉得应该涨价。SET VARIABLE: <<Price += 200>>
我去了公园。SET VARIABLE: <<playersInPark = add(@soul1)>>
我离开了公园。SET VARIABLE: <<playersInPark = remove(@soul1)>>

# 中文格式（等价）
我来了！设定变量：《arrive = true》
结束了。设定变量：《finish = true》

赋值解析流程

  1. 引擎扫描 AI 完整回复，找所有 SET VARIABLE: <<...>> 语句
  2. 逐个解析括号内的赋值表达式，成功的直接操作变量
  3. 解析失败的，存入 SET_VARIABLE 列表
  4. 如果 action 没有声明 resolve 函数，失败赋值直接丢弃并打印警告
  5. 如果声明了 resolve 且在参数中写了 SET_VARIABLE，引擎将列表传给用户处理
操作格式
直接赋值<<VAR = 值>>
加<<VAR += N>>
减<<VAR -= N>>
列表追加<<VAR = add(元素)>>
列表移除<<VAR = remove(元素)>>
字典键赋值<<dict.@key = 值>> 或 <<@actor.attr += N>>

Debug 指南

Debug

编译器报错速查

报错类型原因解决方案
变量未声明action 中使用了未在 vars 中声明的变量在 vars: 中先声明，再使用
文件不存在file:"path" 指向的文件不存在检查路径是否正确，注意相对路径
方法未定义action 中引用了未定义的 memory/context 方法检查方法名是否与定义时一致（大小写敏感）

循环变量名不匹配for/par 变量名与 action 括号中的不一致确保两者完全一致，如都是 @speaker
数据库约束违反数据库操作违反约束引擎原样抛出异常，检查数据库结构
@func out 键不匹配Python 返回字典的 key 与 out 变量不一致确保 key 与 out 声明的变量名一一对应

常见运行时问题

🔎
AI 不输出赋值变量
检查 prompt 中的引导语是否足够明确。建议在 prompt 里写清楚格式：分析完请务必输出 SET VARIABLE: <<VAR = 值>>。也可以用 resolve 函数兼容 AI 的非标准输出。
🔎
AI 角色串台（说了不该说的）
检查 scope 配置。各 action 的 scope 是否正确隔离？比如狼人夜晚发言，scope 应只含狼队成员，不含预言家。
🔎
循环不结束
检查循环退出条件：① for 循环的终止条件是否正确遍历完数组；② module 内的 fork 条件是否正确指向 [OUT]；③ 是否在 module 内用了递归（应避免，见第 17 节）。
🔎
@func 传参为空
检查：① 自动推断模式下 Python 函数参数名是否与 vars 中变量名完全一致；② @actor 变量传入 Python 时收到的是字典，不是字符串，需要用 actor.get("soul") 等方式访问。
🔎
模块局部变量"泄漏"
模块的 vars 在退出后会清空，无法传回母模块。如需让模块向外传递结果，应在 mainflow 层的全局 vars 中预先声明变量，在模块内通过 AI 赋值信号 SET VARIABLE 修改全局变量，而不是依赖模块局部变量。

Debug 技巧

  • 缩小范围：先用简单剧本验证单个 action 行为，再组合进复杂流程
  • 加 @hostgod scope：让一个"上帝"角色进入所有 scope，方便从全局视角审查对话记录
  • 用 super_debug.fems 模板：项目自带的调试剧本，两处并发 + 多轮讨论，适合找隐蔽 bug
  • 随时加人类节点：在关键 action 前后加 @human 节点，随时插话、查看当前状态
  • 检查数据库：所有对话记录都存在数据库中，直接查 soul_scope 字段确认视角隔离是否生效

Core

一个包含 AI 角色、记忆、上下文、并发投票的完整可运行剧本示例：

fems · complete_example.fems
meta:
    name    = "三人聊天室"
    database = file:"chat.db"
    owner   = [1]
    session = new

vars:
    memory_text  = ""
    context_text = ""
    vote_results = {}
    winner       = ""

code:
    memfile = file:"femBridges/MemoryExample.py"
    ctxfile = file:"femBridges/ContextExample.py"
    logic   = file:"vote_logic.py"

actors:
    ai @alice  = soul:1, source:deepseek
    ai @bob    = soul:2, source:glm5
    human @me  = soul:0, source:1

memory rag10(memfile.retrieve_example):
    in: prompt, session_id, @actor
    out: memory_text

context thisSession(ctxfile.findThisSession):
    in: session, @actor
    out: context_text

# ── Actions ──

action alice_speak @ai(@alice):
    prompt: "请简短发言，和大家聊聊今天的话题。"
    scope: [@alice, @bob, @me]

action bob_speak @ai(@bob):
    prompt: "请回应 Alice 的发言，简短即可。"
    scope: [@alice, @bob, @me]

action me_speak @human(@me):
    prompt: "你想说什么？"
    scope: [@alice, @bob, @me]

# 并发投票：Alice 和 Bob 同时投票
action cast_vote @ai(@voter):
    prompt: |
        请投票选出今天最佳发言者（@alice 或 @bob），
        输出：SET VARIABLE: <<vote_results.@voter = @被投票者名>>
    scope: [@voter]
    out: vote_results.@voter(string, "投票目标")

action tally @func(logic.tally_votes):
    in: votes = vote_results
    out: winner

action announce @ai(@alice):
    prompt: "本轮投票结果：{winner} 获得最多票数，恭喜！"
    scope: [@alice, @bob, @me]

mainflow:
    [START] -> me_speak -> alice_speak -> bob_speak -> [VOTE]
    
    # 并发投票，所有人同时投
    [VOTE] -> par @voter in [@alice, @bob]:
        -> cast_vote ->
    -> [tally]: tally
    [tally] -> announce -> [END]
