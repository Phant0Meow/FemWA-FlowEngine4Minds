00

Overview & Design Philosophy

Core

FemWA (Flow Engine for Minds · Work Automata) is a DSL (Domain-Specific Language) and compiler for orchestrating multi-agent workflows. Its core idea: describe AI collaboration the way you write a screenplay.

🖱️
New users: start with the visual editor at femwa.net. It supports drag-and-drop flow building — advanced constructs like for and par have visual equivalents too. No code required. Come back here once you want to understand the syntax in depth.
💡
Three core principles — understand these and you understand Fem:
① Scope = Space: only actors sharing a scope share context — perspective isolation is built-in;
② Vars = Flow Control: variables aren't for passing messages — they drive flow direction;
③ Context is not a variable: LLMs communicate through shared context, not explicit parameters.

Fem files use the extension .fems (the s stands for script). The compiler reads a .fems file, parses the script, and drives agent collaboration according to the rules defined within.

📖
This document describes FemWA v1.0 compiler behavior. Features marked TODO / Not yet implemented are noted — please read carefully.
01

File Structure

Core

A .fems script is organized in the following order. All blocks are optional except mainflow, which is required:

fems · overall structure
meta:       # Script metadata
vars:       # Optional · global state variable pool · must be declared before use
code:       # Optional · external Python module imports · required for @func
actors:     # Recommended · role definitions · AI has no persona if omitted
memory:     # Optional · memory retrieval method definitions
context:    # Optional · context extraction method definitions
action ...: # Required · action definitions · at least one, can be many
module ...: # Optional · module definitions · can be many
mainflow:   # Required · main flow orchestration
⚠️
Indentation-sensitive: indentation expresses ownership, similar to Python. Newline-sensitive: flow control constructs like fork, for, and par must follow strict indentation and newline rules — the parser will error otherwise.

Conventions & Symbol Reference

Core
  • Case-sensitive: variable names and action names are case-sensitive
  • Comments: use # or // inline
  • @ symbol: actor entity names always carry @ — in definitions, references, and parameter passing
  • File paths: must use file:"path/to/file" format; bare text is treated as a string literal
  • Multi-line text: use | followed by an indented newline for multi-line string literals

meta — Script Metadata

Core

Defines the script's basic properties and runtime environment.

fems
meta:
    name     = "My Werewolf Game"
    version  = 1.0
    owner    = [1, 2]
    database = file:"werewolf.db"
    session  = new           # or specify an integer session_id
    system_safety = |
        No database deletion. No harmful content generation.
    output_style = "Keep replies concise and professional."
FieldTypeDescription
nametextScript name, written to the database when a new session is created
ownerinteger arrayOwner user_ids; the compiler auto-injects user info into context
databasefile pathSQLite database path; defaults to ./chronica.wor if not specified
sessioninteger / newnew creates a new session automatically; a number loads an existing one; 17/new is also valid
system_safetytext / file pathSafety policy injected into all actors' context automatically
output_styletext / file pathOutput quality guidelines, auto-injected into context
ℹ️
session = 17 / new means: load session 17 if it already exists; if you happen to be creating it fresh and it becomes 17, create it automatically; otherwise the backend will ask whether to proceed.
04

vars — Global Variable Pool

Core

All global state variables are declared here. Variables must be declared before use — directly assigning an undeclared variable causes an error.

fems
vars:
    turn_count      = 1
    game_over       = false
    alive_players   = [@seer, @wolf]        # @actor array
    agent_locations = {@ellis: "bedroom", @bob: "park"}  # dict
    @speaker        = ""                  # @actor-type variable, empty value is ""
    hp              = {@wolf: 100, @seer: 100}

Key Points

  • @actor-type variables: variable name must start with @, e.g. @speaker, to be recognized as an actor type
  • @actor dual-perspective access: hp.@wolf and @wolf.hp are fully equivalent (see section 19)
  • Dictionary default: using {dict[key]} in a prompt where the key doesn't exist resolves to an empty string — no error
05

actors — Role Definitions

Core

Uses a strong-typed declaration style: type @name = attributes.

fems
actors:
    ai @hostgod = soul:0, source:deepseek
    ai @wolf    = soul:1, source:glm5.1, tools:[deep_think]
    ai @seer    = soul:2, source:glm5
    human @player = soul:9, source:0  # source:0 = human user without admin perspective
FieldDescription
soul:IDUnique soul ID, mapped to the database souls table; the compiler extracts the system prompt automatically
source:modelFor AI: the model name; for humans: the numeric user ID; omit for random assignment
source:0Special value: a human user without admin perspective — only sees through the soul's viewpoint (ideal for Werewolf-style players)
tools:[...]Attached tool list, e.g. web_search, deep_think
🚫
Actor entity names always carry @ and must never be stripped: in definitions ai @ellis = ..., in references @ai(@ellis), in variable passing @speaker = @ellis. The @ is part of the variable name — without it the actor type cannot be recognized.
06

code — External Python Modules

Core

Import Python files for use by @func and resolve. Paths must use the file:"..." format. Absolute paths are preferred; a relative path like file:"wait.py" resolves to /Users/{user_name}/FemWA/func_code/wait.py.

fems
code:
    game_logic = file:"utils/game.py"
    dev_ops    = file:"utils/deploy.py"
    memfile    = file:"femBridges/MemoryExample.py"
⚠️
The following are reserved keywords and cannot be used as code aliases: meta, vars, code, actors, action, module, flow, mainflow, memory, context
07

memory & context Definitions

Core

memory and context methods define how AI actors receive memory and conversational history. They live at the same level as actions and modules.

fems · definition syntax
memory rag10(memfile.retrieve_example):
    in: prompt, session_id, @actor
    out: memory_text

context thisSession(ctxfile.findThisSession):
    in: session, @actor
    out: context_text

Reserved in Parameters

FieldValue Passed
promptThe current action's prompt after variable substitution
@actorThe current speaker's actor_info dict (Python-side param name auto-maps to actor_info)
session / session_idThe current session ID
turn / turn_idThe current turn ID
Any other nameFetched from global vars by the same name

Referencing in an Action

fems
action speak @ai(@ellis):
    prompt: "Hello"
    memory:  rag10        # omit to use the first memory method, or the built-in default
    context: thisSession  # omit to use the first context method, or the built-in default
ℹ️
If no context method is defined, the FEM compiler uses a built-in default — it retrieves context from the current session filtered by scope. This is sufficient for most scenarios.
08

AI Action — @ai

Core

The most common action type. Calls an LLM to complete one round of dialogue.

fems · full field reference
action wolf_kill @ai(@wolf):
    prompt: |
        It is night {turn_count}. Surviving players: {alive_players}.
        You are the werewolf. Choose a target to eliminate.
        After reasoning, output: SET VARIABLE: <<@KILL = @player_name>>
    scope: [@hostgod, @wolf]
    out: wolf_target(string, "kill target")
    resolve: game_logic.resolve_target(@KILL, alive_players)
    max_retries: 3
    fallback: host_emergency
    context: thisSession
    memory: rag10

Field Reference

prompt

Supports {var} f-string interpolation. By default all global variables are substituted; if an in: field is present, only the listed variables are substituted (useful for controlling information exposure).

fems · in field example
action coder_act @ai(@coder):
    in: my_task = task_list[@coder]   # left=name used in prompt, right=global variable expression
    prompt: "Your task is: {my_task}"    # {other_var} won't be replaced — preserved as-is

scope

Defines which actors can see this round of dialogue. Supports dynamic variables, e.g. scope: playersInPark. See section 18 for details.

out — AI Output Variable

Declares which variable the AI must output. Format: variable_name(type, "label"). The AI must include SET VARIABLE: <<var_name = value>> in its reply. The variable must already be declared in vars.

Writing to a dictionary key is also supported: vote_results.@voter(string, "") stores the value into the specified key of the dict.

resolve — Validation Function (optional)

Used when complex validation or non-standard AI output format handling is needed.

fems + python · resolve example
# in .fems
resolve: game_logic.resolve_target(@KILL, alive_players, SET_VARIABLE)
# including SET_VARIABLE causes the compiler to pass in any failed-parse assignment list

# Python (utils/game.py)
def resolve_target(@KILL=None, alive_players=None, SET_VARIABLE=None, **kwargs):
    if @KILL and @KILL in alive_players:
        return [(True, False, "")]           # (is_success, show_to_ai, feedback)
    if SET_VARIABLE:                           # built-in parse failed — handle manually
        import re
        for item in SET_VARIABLE:
            m = re.match(r'@KILL\s*=\s*@(\w+)', item)
            if m and f"@{m.group(1)}" in alive_players:
                return [(True, False, "")]
    return [(False, True, "Target not in alive players. Please choose again.")]
⚠️
Current version limitation: max_retries and fallback fields are reserved but auto-retry is not yet implemented. The show_to_ai and feedback return values in resolve are also not yet active.

Dynamic @actor_expr

ℹ️
When using a dynamic variable in the action's parentheses (e.g. @ai(@speaker)), the for/par loop variable name must exactly match the variable name in the parentheses — otherwise the engine errors immediately.

Human Action — @human

Core

Syntax is largely identical to @ai. Used to incorporate human player input into the workflow.

fems
action human_vote @human(@player):
    prompt: Cast your vote:
    scope: [@hostgod] + alive_players
    out: vote_target(dropdown, choices=alive_players, label="Vote")
    resolve: game_logic.resolve_human_vote

# Let a human play as a specific AI role, inheriting that role's scope:
action seer_act @human(@player) as (@seer):
    prompt: You are now the Seer. Who do you want to inspect?
    scope: [@seer, @hostgod]
  • The out type dropdown maps to a dropdown UI component in the frontend
  • Human actions support context (same syntax as AI actions) but not memory
  • as (@role): allows the human player to enter a specific AI role's private perspective

Assignment Action — @assign

Core

Lightweight assignment — does not call an LLM, directly modifies global variables.

fems · supported operations
action next_turn @assign:
    out: day_count += 1

action reset_scene @assign:
    out: current_scene = "tavern", time = sunset   # multiple assignments, comma-separated

action update_park @assign:
    out: playerInPark   = add(@Alice)
         playerInMarket = remove(@Alice)   # multi-line out is supported
OperationMeaning
= valueDirect assignment
+= NAdd N
-= NSubtract N
= add(element)Append element to list
= remove(element)Remove element from list
11

Python Function Call — @func

Core

Calls a Python function declared in the code: block to handle complex logic.

fems
action resolve_night @func(my_utils.resolve_night):
    in:
        kill_target = wolf_target        # left=Python param name, right=FEM variable expression
        save        = witch_save_target
    out: dead_tonight, alive_players

in Parameter Rules

  • Explicit mode (with in:): strictly passes declared params, format param_name = variable_expression
  • Auto-inference mode (no in:): matches Python function signature against vars by name

out Receiving Rules

Function Return Typeout Behavior
Returns a dictMatches by key to variable names in out — keys must exactly match
Returns a single valueWhen out declares exactly one variable, assigned directly
out: dict.key formFunction returns just the final value; the compiler handles writing it to the specified dict key

Passing @actor Variables to Python

When an in variable is an @actor type, the engine automatically packages it as a dict for Python:

python · received actor dict
{
    "type":     "ai",
    "name":     "@ellis",
    "soul":     3,
    "hp":       100,         # from the hp dict in vars
    "location": "tavern"    # from the location dict in vars
}
⚠️
Common mistake: when using out: dict.key, the function should return only the final value — don't build the dict yourself. When returning a dict, all keys must exactly match the declared out variable names — extra or missing keys will error.
12

Basic Chain Flow

Flow

Flow chains Nodes (position markers) together. [START] and [END] are reserved keywords and must exist in mainflow.

💡
Key distinction: Action = behavior (what to do, has no position); Node = a position container (holds an action or module); Flow = a directed graph that chains Nodes together.

Reference an Action without prefix; reference a Module with the & prefix.
fems · sequential chain
[START] -> action0 -> [B]:action1 -> &module1 -> [END]

# Split across multiple lines when it gets long — connect head and tail nodes:
[START] -> wolf_discuss -> seer_check -> [A]:tell_seer
[A] -> witch_ask -> resolve_night -> [END]

Loops Require Nodes

❌ Not a loop
# This just runs action1 a second time
action1 -> action2 -> action1
✅ A real loop
# Returns to [A]'s position and continues from there
[A]:action1 -> action2 -> [A]
13

Concurrent Fork

Flow

fork declares concurrent branches at this point. Each branch runs in a new child thread — ideal for calling multiple LLMs simultaneously.

fems · two syntaxes
# Syntax 1: indented branches
[A] -> [B] ->
fork:
    -> [visit]
    -> [solo]

# Syntax 2: multiple same-indent lines (equivalent)
[C] -> [visit]
[C] -> [solo]

# With conditions — if (Python expression, @actor auto-converted to dict)
[A] ->
fork:
    -> if (var == true) -> [B]
    -> if (score > 5 and level >= 3) -> [C]
    -> if (@Portia.hp == 0) -> [D]   # @actor expressions work directly
ℹ️
fork only handles splitting, not merging. Use join (next section) for merging. Branches without if conditions all launch unconditionally in parallel.
💡
You can skip fork entirely and use the mermaid-style multi-line arrow notation to describe branches — sometimes it's simpler. The benefit of fork: less typing when there are many branches, and a cleaner structure for debugging.

Merge with Join

Flow

join is defined at the receiving end, declaring "which upstream nodes I'm waiting for."

fems
join(all):
    [B] ->
    [C] ->
to [next_node]

# join(2): continue once 2 upstream nodes arrive; cancel remaining branches
join(2):
    [D] ->
    [E] ->
    [F] ->
to [next_node]
  • join(all): wait for all declared upstream nodes to arrive
  • join(N): continue once N upstream nodes arrive; cancel remaining branches (only those from this fork)

join and Mermaid-style Syntax Are Equivalent

fems · two equivalent styles
# join style
join(all):
    [B] ->
    [C] ->
to [next_node]

# Equivalent mermaid-style
[B] -> [next_node]
[C] -> [next_node]
💡
You can skip join entirely: pointing multiple branches at the same node in mermaid style naturally merges them — equivalent to join(all). Only write join(N) when you need the "stop remaining branches after N arrive" behavior.
15

Sequential Loop — for

Flow

Iterates over an array, executing the loop body sequentially for each element — equivalent to an unrolled sequential chain. The entire for block is a single unit; the preceding node is written on the same line as the for keyword.

fems · complete for block
# Full structure: preceding node on the same line as for
[B] -> for @speaker in speaker_array:
    -> [C]:speak -> [D]:respond ->
-> [END]

# Loop variable need not be @actor — plain variables work too
[A] -> for i in [1, 2, 3]:
    -> tick_action ->
-> [END]

# for + if: route by type (exit node is optional)
[B] -> for @player in alive_players:
    -> if (@player.type == "ai")    -> ai_speak ->
    -> if (@player.type == "human") -> human_speak ->
-> [D]
💡
Body markers: inside a loop, mark the first node with a leading -> and the last node with a trailing ->. The exit node is optional — if provided, flow continues there after the loop ends; if omitted, the branch terminates automatically. Removing these leading/trailing arrows, the loop body syntax is identical to regular flow.
🚫
If the loop variable is an @actor type, its name must exactly match the dynamic variable name in the action definition's parentheses — otherwise the engine errors immediately to prevent silently using the wrong actor.

Animation: for sequential loop

Parallel Loop — par

Flow

A shorthand for fork + join: iterates over an array, executing concurrently for each element. Dramatically speeds up parallel LLM calls. The preceding node is written on the same line as the par keyword.

fems · par examples
# Example 1: preceding node on same line, concurrent module calls
[A] -> par @coder in coders:
    -> &CoderSandbox(@coder, task_list[@coder]) ->
-> [D]

# Example 2: each branch executes multiple steps concurrently
[before] -> par @coder in workers:
    -> [A]:writecode -> [B]:trycode ->
-> [D]

# Variable need not be @actor
[A] -> par i in [1, 2, 3]:
    -> tick_action ->
-> [END]

# Exit node is optional — branches terminate automatically when done
[VOTE] -> par @player in alive_players:
    -> cast_vote ->
# No exit node — each branch stops after finishing
  • Multiple nodes inside a branch run sequentially; branches run concurrently with each other
  • The parallel variable is branch-local — different values per branch, no interference; referenced directly in actions without explicit passing
  • Body markers are the same as for: leading -> marks the first node, trailing -> marks the last
  • Exit node is optional — if present, all branches converge there; if absent, each branch terminates independently
  • The parallel variable need not be @actor — par i in [1,2,3]: is fully valid
  • If the parallel variable is an @actor type, its name must exactly match the action definition's dynamic variable name — otherwise the engine errors immediately

Modules

Advanced

A Module is a black-box sub-workflow with an entry point ([IN]), exit point ([OUT]), break exit ([BREAK]), local vars, and an internal flow.

fems · module definition
module CoderSandbox(task_var):
    vars:
        finish = false         # local variable — cleared when module exits
    
    action write_code @ai(@coder):
        prompt: "Task: {task_var}. Use shell tools to write the code."
        scope: [@coder]
    
    action submit_code @ai(@coder):
        prompt: |
            Test the code. Once confirmed correct, output:
            SET VARIABLE: <<finish = true>>
        out: finish(bool, "done?")
    
    flow:
        [IN] -> write_code -> submit_code ->
        fork:
            -> if (finish == true)  [OUT]
            -> if (finish == false) [IN]

Calling a Module

fems
# In mainflow or another module's flow
[START] -> plan -> &CoderSandbox(task_var) -> [END]

# Combined with par for concurrent module instances
[START] -> par @coder in coders:
    -> &CoderSandbox(task_list[@coder]) ->
⚠️
Module local variables are cleared on exit and cannot be returned to the parent module. Use global variables to pass results out.
Avoid module recursion: steps inside a module count as 1 globally — recursion creates infinite loops that cannot be caught by max_steps.
18

scope — Perspective Isolation

Advanced

scope defines "which actors can see this round of dialogue" — the core mechanism for perspective isolation in Fem.

fems · scope usage
# Static scope: only these two actors can see this
scope: [@hostgod, @wolf]

# Dynamic scope: variable resolved to @actor list at runtime
scope: playersInPark

# Mixed
scope: [@hostgod] + alive_players

When storing a dialogue record, the system writes the soul IDs of all actors in scope to the soul_scope database field. FEM's built-in context method filters by this field, so each actor can only retrieve the context they were present for.

💡
A useful analogy: scope is like a "room" — only people present can remember the conversation. More precisely: a god stores all history in a cosmic hard drive, but only those who were actually there are authorized to retrieve it. This is why perspective isolation in a Werewolf game requires just one line of scope.
19

@actor Type System

Advanced

An Actor is not just a role configuration — it is a variable type with identity and state, assignable and passable like any variable.

Dual-Perspective Attribute Access

The dict-perspective and entity-perspective are fully equivalent — use whichever reads more naturally:

fems · equivalent forms
# Assume vars contains: hp = {@wolf: 100, @seer: 100}

hp.@wolf        # dict perspective: look up @wolf in the hp registry
@wolf.hp        # entity perspective: @wolf's hp attribute
# Fully equivalent — usable in prompts, assignments, and if conditions

# Prompt f-string
"Your current HP is {hp.@wolf}"      # same as
"Your current HP is {@wolf.hp}"

# AI assignment signal
SET VARIABLE: <<hp.@wolf -= 30>>   # same as
SET VARIABLE: <<@wolf.hp -= 30>>

Static Reserved Attributes

When resolving @entity.attribute, the following 5 reserved attributes are checked first. If not matched, the runtime looks for a same-named dict in vars:

type
Actor type: "ai" or "human"
soul
Soul ID (integer)
source
Model name or user ID
tools
Attached tool list
name
Actor name as string

This means you can attach dynamic state to actors without modifying their definition — just create a matching dict in vars and @entity.attribute access works automatically.

AI Variable Assignment Signal

Advanced

Guide the AI through the prompt to output assignment signals in its reply, driving updates to flow-control variables.

AI output examples (multiple valid formats)
# Standard format
I vote for player 3. SET VARIABLE: <<@VOTE_TARGET = @player3>>
I think prices should rise. SET VARIABLE: <<Price += 200>>
I went to the park. SET VARIABLE: <<playersInPark = add(@soul1)>>
I left the park. SET VARIABLE: <<playersInPark = remove(@soul1)>>

Assignment Parsing Flow

  1. The engine scans the full AI reply for all SET VARIABLE: <<...>> statements
  2. Parses each assignment expression — successful ones update the variable immediately
  3. Failed parses are stored in the SET_VARIABLE list
  4. If the action has no resolve function, failed assignments are discarded with a warning
  5. If resolve is declared and SET_VARIABLE appears in its parameters, the engine passes the list for user handling
OperationFormat
Direct assign<<VAR = value>>
Add<<VAR += N>>
Subtract<<VAR -= N>>
Append to list<<VAR = add(element)>>
Remove from list<<VAR = remove(element)>>
Dict key assign<<dict.@key = value>> or <<@actor.attr += N>>

Debug Guide

Debug

Compiler Error Reference

Error TypeCauseFix

Common Runtime Issues

🔎
AI doesn't output the assignment variable
Check that the prompt's instruction is explicit enough. Recommend stating the exact format in the prompt: After your analysis, you must output SET VARIABLE: <<VAR = value>>. You can also use a resolve function to handle non-standard AI output.
🔎
AI actor says things it shouldn't (perspective leak)
Check scope configuration. Is each action's scope correctly isolated? For example, werewolf nighttime speech should have scope limited to wolf team members — excluding the Seer.
🔎
Loop never ends
Check exit conditions: ① Is the for loop's termination condition correctly iterating through the array? ② Does the module's fork condition correctly point to [OUT]? ③ Is there recursion inside a module? (Should be avoided — see section 17.)
🔎
@func receives empty parameters
Check: ① In auto-inference mode, do the Python function param names exactly match variable names in vars? ② When an @actor variable is passed to Python, it arrives as a dict, not a string — access fields with actor.get("soul") etc.
🔎
Module local variables "leaking"
Module vars are cleared on exit and cannot be returned to the parent. To pass results out of a module, pre-declare a variable in the mainflow-level global vars, then update it from inside the module via the SET VARIABLE signal — don't rely on module-local variables for output.

Debug Tips

  • Narrow scope: validate individual action behavior with a simple script first, then compose into complex flows
  • Add @hostgod to all scopes: gives one "god-mode" actor access to all scopes, making it easy to review the full dialogue history
  • Use the super_debug.fems template: the project's built-in debug script features two concurrent branches and multi-turn discussion — ideal for tracking subtle bugs
  • Insert human nodes anywhere: add a @human node before or after key actions to pause, interject, or inspect current state
  • Inspect the database: all dialogue records are stored in the database — query the soul_scope field directly to confirm perspective isolation is working

Complete Example

Core

A fully runnable script featuring AI actors, memory, context, and a concurrent voting round:

fems · complete_example.fems
meta:
    name     = "Three-Person Chat Room"
    database = file:"chat.db"
    owner    = [1]
    session  = new

vars:
    memory_text  = ""
    context_text = ""
    vote_results = {}
    winner       = ""

code:
    memfile = file:"femBridges/MemoryExample.py"
    ctxfile = file:"femBridges/ContextExample.py"
    logic   = file:"vote_logic.py"

actors:
    ai @alice  = soul:1, source:deepseek
    ai @bob    = soul:2, source:glm5
    human @me  = soul:0, source:1

memory rag10(memfile.retrieve_example):
    in: prompt, session_id, @actor
    out: memory_text

context thisSession(ctxfile.findThisSession):
    in: session, @actor
    out: context_text

# ── Actions ──

action alice_speak @ai(@alice):
    prompt: "Share a brief thought on today's topic with the group."
    scope: [@alice, @bob, @me]

action bob_speak @ai(@bob):
    prompt: "Respond briefly to Alice's message."
    scope: [@alice, @bob, @me]

action me_speak @human(@me):
    prompt: "What would you like to say?"
    scope: [@alice, @bob, @me]

# Concurrent voting: Alice and Bob vote simultaneously

    prompt: |

Async Engine Architecture

Dev

AsyncEngine (located at femCompiler/femAsync.py) is the hub of the entire concurrent system — managing all executors, scheduling coroutine tasks, and providing unified async primitives.

Core Idea: Scheduling and Execution Are Decoupled

The engine follows a "single-threaded brain, delegated heavy lifting" model:

  • asyncio event loop: the sole scheduler, single-threaded, orchestrates all Agent coroutines — no operation blocks the event loop
  • Heavy tasks (CPU-intensive, synchronous I/O, human input waiting): always delegated to independent execution pools (process pool or thread pool); the main loop only awaits results

This separation keeps the main loop always responsive — even if one branch is running a long computation or waiting for an LLM, other branches continue advancing.

Three Executor Types

Process Pool
CPU-intensive tasks. Size auto-adapts to CPU core count. Tasks queue automatically — when all workers are busy, subsequent tasks wait in an internal queue without being dropped.
I/O Thread Pool
Synchronous blocking I/O (LLM API calls) and @func user code. Each thread launches a persistent background event loop at creation time, completely eliminating no running event loop errors.
Human Input Manager
Thread-safe async waiting built on threading.Event. A coroutine calling wait_for_input suspends; the external side wakes it via provide_input. Supports timeouts.

Task Scheduling and Join Modes

Each Agent branch (fork branch) is an asyncio.Task. At fork time, engine.create_task() creates and auto-tracks tasks; at join time, gather_with_cancel waits or cancels them uniformly.

Join ModeBehavior
allWait for all branches to complete
anyCancel all others once any one completes
nCancel all others once the specified count complete

Context Isolation

Python's standard library contextvars automatically isolates context variables for each coroutine/thread. Each Agent branch has an independent ExecutionContext — internal variables like node_name and loop variables don't interfere with each other. No manual locking required.

Runtime Integration

Dev

FEMRunner (located at femCompiler/FEM_runtime.py) is the workflow executor. It holds an AsyncEngine instance and delegates all concurrency needs to it.

Node Execution Routing

Node TypeExecution Method
@funcengine.run_in_thread, executed in the thread pool (may auto-switch to process pool based on complexity in the future)

Flow Control Primitive → Engine Mapping

Flow SyntaxRuntime Implementation
Fork_run_fork creates an engine.create_task() for each branch, stored in the _join_tasks dict
Join_run_join retrieves the task list and calls engine.gather_with_cancel with the configured join mode

Main Flow Entry Point

FEMRunner.run() is the synchronous entry point. It detects whether an event loop is already running:

  • CLI mode (no event loop): calls asyncio.run(self.run_async()) to start a new loop
  • FastAPI mode (loop already running): directly await runner.run_async()

Troubleshooting

🔎
Frontend receives no node info (node_name is empty)
The main flow didn't create a top-level ExecutionContext. Ensure run_async is wrapped in with ExecutionContext("__main__"):.
🔎
Flow silently exits with no errors
Entry inference failed (flow.entry is empty, or all nodes have in-edges). Check whether the workflow explicitly declares a [START] node.
🔎
AI calls report no running event loop
Confirm AsyncEngine's thread pool is configured with an initializer (to start a background event loop for each thread). This is built-in engine behavior and normally requires no extra config — if this error appears, the engine initialization may have been bypassed.
🔎
All fork branches execute unconditionally
_run_fork is not evaluating condition edges. Confirm the branch entry collection logic calls self._eval_condition(e.condition) to filter out false conditions.

Scaling from Single Machine to Distributed

The current architecture already decouples the scheduling layer (FEMRunner / AsyncEngine) from the execution layer (thread pools, process pools), making distributed scaling natural:

  • Scheduler cluster: deploy multiple stateless FEMRunner instances behind a load balancer
  • CPU Worker cluster: route CPU-intensive @func nodes to dedicated Python Workers via Redis Streams or RabbitMQ
  • GPU inference cluster: call GPU inference nodes via HTTP/gRPC to deployed model services (e.g. vLLM)
  • Human input management: swap HumanInputManager's backend to Redis Pub/Sub to support cross-machine input waiting and wake-up
